# The Showcase Manifest Contract

> **This repo is the source of truth for the whole gallery.** The root `index.html` and every
> per-item detail page are **generated** by `build/build_showcase.py` from the manifests in
> `manifests/*.json`. **Do not hand-edit `index.html` or `detail/*.html`** — a rebuild overwrites
> them. To change what the gallery shows, edit a manifest and re-run the generator.
>
> Root cause of the old gallery's inconsistency: the root index was hand-edited on the box with no
> generator and no schema, so every curator produced a differently-shaped card and nothing survived
> a rebuild. This contract fixes that structurally.

## How to add or change an item

1. Drop a `manifests/<slug>.json` conforming to the schema below (validate against
   `manifests/_schema.json`).
2. Put its thumbnail + screenshots under `assets/<slug>/`. To capture a live standalone item as a
   card thumbnail, use `build/shot_urls.js` (a JSON list of `{url, out}`); items serve directly over
   the Tailscale mirror at `http://100.74.37.45:8090/<slug>/` with no Access gate. A backend-off
   item (its `standalone` is `false` and its server is down) has no real screenshot — omit
   `thumbnail` and the generator renders an honest title placeholder. **Never fabricate a screenshot.**
3. Run `python build/build_showcase.py`. It validates every manifest (failing loudly on a bad one),
   then regenerates `index.html` + `detail/<slug>.html` + syncs `showcase.css`.
4. Commit, then deploy with `bash build/deploy.sh`. It rebuilds, backs up the box `index.html`, and
   `scp`s ONLY the generated artifacts (`index.html`, `showcase.css`, `detail/`, `assets/`) to the
   box — leaving every item application directory untouched. Source of truth is this git repo; the
   box (and GitHub Pages) are deploy targets. When the box retires, only `BOX_HOST`/`BOX_DIR` in
   `deploy.sh` change.

The matchmaker item is special: its whole contribution (manifest, bespoke `custom_detail` page,
screenshots, reading cards) is generated in the sibling `big-five-to-philosopher-matchmaker` repo
under `showcase/` and staged here by that repo's `showcase/publish.py`. Re-run that to refresh it.

The generator **never touches item application directories** (`/big-five-matchmaker/`,
`/yuyu-linguistic-risk-detector/`, `/foundry/`, …). It writes only the root `index.html`, the
`detail/` pages, `showcase.css`, and `assets/`. Detail pages live under `detail/<slug>.html`
precisely so they never collide with an app served at `/<slug>/`.

## The `foundry/` exception

`foundry/` is generated by its own long-standing tool (`ops/box/stage-projects.py`, a systemd
timer that harvests foundry project blurbs from `IDEA.md` + git). We do **not** subsume it. It
appears in the gallery as a **single manifest entry** (`manifests/foundry.json`) whose `open` link
points at the foundry sub-gallery. Its internals remain its own.

## Manifest schema (per item)

```jsonc
{
  "slug":         "big-five-matchmaker",     // REQUIRED. url-safe; matches assets/<slug>/ and detail/<slug>.html
  "title":        "The Philosopher Matchmaker",  // REQUIRED. card + detail <h1>
  "tagline":      "Give it a Big-Five profile; it ranks philosophers you'd resonate with — and shows the number behind every reason.",  // REQUIRED, <= 140 chars. The one-line hook on the card. NOT an essay.
  "category":     "standalone",   // REQUIRED. one of: flagship | backend | standalone | prototype | ops
  "badges":       [{"label": "standalone", "kind": "ok"}],  // kind: ok|backend|ext|flag|art|soon
  "thumbnail":    "assets/big-five-matchmaker/thumb.png",   // card image. 16:10. REQUIRED for a scannable gallery.
  "screenshots":  ["assets/big-five-matchmaker/shots/a.png", "..."],  // detail carousel. optional.
  "functionality":[                                 // detail "what it does" menu. optional.
    {"name": "Temperament resonance", "blurb": "Cosine over 30 disposition facets."}
  ],
  "links": {
    "open":         "./big-five-matchmaker/",       // REQUIRED. where "open it" goes (the live app)
    "how_it_works": "https://claude.ai/code/artifact/..."  // optional external deep-dive
  },
  "detail_body":  ["<p>full essay paragraph…</p>", "..."],  // the essay that used to bloat the card. HTML paragraphs. Rendered on the generated detail page.
  "custom_detail":"detail-src/big-five-matchmaker.html",    // optional. if set, the generator COPIES this pre-built page to detail/<slug>.html INSTEAD of generating one (for bespoke pages like the elevated matchmaker). Path is repo-relative.
  "ops_note":     "The app browses standalone; live 'scan your own text' needs the uvicorn backend on the box.",  // optional amber ops caveat
  "substance_rank": 90,   // REQUIRED. higher = earlier within its category section. flagship/backend/standalone/prototype are the section order; rank sorts within.
  "standalone":   true    // REQUIRED bool. true => "runs in your browser". false => "needs a backend on". Drives the standalone-vs-backend filter.
}
```

### Field rules the generator enforces

- `tagline` **must be <= 140 chars** — the build fails loudly if longer. This is what keeps cards to
  one glance; the essay goes in `detail_body`, never the tagline.
- `category` and `standalone` are independent: the flagship is `standalone:false` (its live scan
  needs a backend) but `category:flagship`. The filter reads `standalone`; the section reads
  `category`.
- Exactly one item may be `category:flagship`; the generator warns if more.
- `custom_detail` and `detail_body` are mutually exclusive — a custom page owns its whole body.
